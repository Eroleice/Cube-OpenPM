# OpenCode 执行规则

OpenCode 是默认唯一允许修改项目源码的 runtime。

## 基础规则

- 修改代码前必须读取来源 task。
- 必须保持在 task 范围内。
- 必须以当前仓库状态作为事实来源。
- 执行前必须识别当前 TypeScript 项目的包管理器、可用脚本、测试框架、构建入口和既有目录结构。
- 修改 TypeScript 或构建敏感代码后，应运行项目已有的相关检查；如果缺少对应脚本，必须在终端输出中说明无法运行的原因和替代验证。
- 不要创建、修改或覆盖 `.openpm/progression/04.coding/` 下的 coding report；OpenPM 会在 OpenCode 退出后根据 stdout、stderr 和仓库状态统一生成同名 coding report。
- 除非未来有明确 OpenPM 命令要求，否则不要 commit。
- 不要修改无关的用户改动。

## TypeScript 编码规则

- 所有代码必须通过 TypeScript strict mode，不得以关闭 strict、放宽 tsconfig 或跳过类型检查的方式完成任务。
- 非必要不得使用 `any`。确实必须使用时，必须在相邻代码注释或最终报告中说明原因、影响范围和后续收敛方式。
- 优先使用 `unknown`、类型守卫、zod schema、discriminated union 和显式类型收窄替代 `any`、盲目类型断言或宽泛 object。
- 不得使用 `// @ts-ignore`。确实无法避免的临时兼容只能使用 `// @ts-expect-error`，并必须在注释中说明触发原因和移除条件。
- 避免非空断言 `!`。如果使用，必须能从生命周期、构造流程或前置校验证明值一定存在。
- 避免双重断言，例如 `value as unknown as TargetType`。如果边界数据无法直接证明类型，必须先做 runtime 校验或显式转换。
- 导出的函数、复杂领域方法、公共 API、use-case、repository、facade、IPC handler、network handler 必须显式声明参数类型和返回类型。
- 模块内部短小、类型可明显推断的局部变量可以依赖类型推断；跨模块边界不得依赖隐式结构。
- 外部输入、文件读取、配置表、存档、IPC、network message、URL/query、用户输入和工具输出必须先通过 zod 或明确类型守卫校验，不得直接类型断言后使用。
- 配置、存档、DTO、事件、IPC message、network message 必须有稳定命名的 TypeScript 类型；跨进程、跨网络或持久化边界还必须有 zod schema。
- 领域层状态优先使用明确的 readonly 数据结构、品牌化 ID、discriminated union 和不可变更新，不要使用松散 string/object 混合结构表达状态。
- 对枚举型状态，优先使用 string literal union 或 `as const` 映射；除非需要运行时反向映射，不要默认使用 TypeScript enum。
- 错误处理优先使用项目共享的 Result/AppError 模式或明确的错误类型；不要在领域规则中抛出无法被调用方分类处理的裸字符串错误。
- 异步函数必须明确错误边界。Repository、IPC、network、文件系统和资源加载代码不得吞掉异常；需要降级时必须返回可观察的错误结果。
- 不要用可选链和默认值掩盖非法状态。业务必需字段缺失时应在边界校验阶段失败，而不是在深层逻辑中静默兜底。
- 不要把大型函数、跨职责 helper 或临时脚本式流程堆在单个文件里。单文件职责应清晰，复杂功能应拆成领域规则、应用编排、运行时适配和表现层适配。
- 不要跨层直接导入实现细节。上层通过 facade、use-case、DTO、event、schema 或接口协作，不得绕过层级边界调用内部模块。
- 领域规则、状态转换、权限判断、数据校验和持久化迁移代码必须保持纯 TypeScript、确定性、可单元测试；不得直接依赖 DOM、具体 UI runtime、平台 runtime 实例、真实时间、随机数或全局可变状态。
- 需要时间或随机性的领域逻辑必须通过显式参数、seed、clock 或 RNG 接口注入，确保测试可复现。
- 公共类型、schema 和常量命名必须稳定、语义明确；不要使用 `data`、`payload`、`obj`、`info` 这类含义不清的导出名称。
- 文件和目录命名应体现模块职责。不要创建 `utils.ts`、`helpers.ts` 这类泛化垃圾桶文件；如需工具函数，应按领域或能力命名，例如 `inventoryMath.ts`、`saveMigration.ts`。
- 导入方向应保持清晰：领域层不得导入应用层、表现层、运行时适配或平台层。
- 实现完成后，如果项目提供 typecheck 脚本，必须运行对应 typecheck；涉及构建入口、配置或导出边界时还必须运行对应 build。常见命令包括 `npm run typecheck`、`pnpm typecheck`、`npm run build` 或项目已有等价命令。

## 代码注释规则

- 代码注释默认使用中文；框架名、库名、协议名、API 名、类型名、专有概念可以保留英文。
- 每个代码文件顶端必须有注释块，说明文件主要功能、所属模块、所属架构层级或资源管线，以及典型调用入口。
- 文件顶端注释必须帮助读者判断该文件能改什么、不能改什么，所属边界。
- 导出的函数、类、类型、schema、常量、use-case、facade、repository、IPC handler、network handler 必须有中文注释说明用途和调用方式。
- 复杂的非导出方法、状态机、公式、迁移、资源加载流程、事件派发流程也必须有中文注释；简单局部变量和自解释代码不需要机械注释。
- 方法注释必须说明：用途、调用时机或调用方、关键参数含义、返回值含义、错误或失败场景。
- 参数如果会影响程序用法、性能、UI、窗口、资源加载、存档、网络或平衡，必须说明作用、影响范围和建议调整范围或合法取值范围。
- 常量目录下每个导出参数必须有完整中文注释，至少包含用途、影响范围、建议调整范围或合法取值范围；不得只写变量名或一句模糊说明。
- 对算法、公式、平衡曲线、掉落权重、战斗判定、时间推进、随机数和存档迁移，必须注释设计意图和修改风险。
- 对跨层调用必须注释为什么通过当前 facade、DTO、event、schema 或接口协作，避免后续维护者绕过层级边界。
- 对临时兼容、技术债、`@ts-expect-error`、必要的类型断言、必要的非空断言，必须写清原因、影响范围和移除条件。
- 注释应该解释业务意图、架构边界和调参影响，不要写“给变量赋值”“调用函数”这类重复代码字面意思的低价值注释。
- 当修改已有文件时，如果文件缺少顶端注释或被改动的主要方法缺少说明，应同步补齐相关中文注释。
- 最终报告的“修改文件”或“交接备注”中必须说明是否补充或更新了关键注释；如果没有补充且涉及代码修改，必须说明原因。
