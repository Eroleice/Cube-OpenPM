# 002.milestone-2

- 来源：DESIGN.md + MSC.md

## 目标

交付用户直接操作魔方的核心交互闭环：右键拖动只改变观察视角，左键拖动在有效条件下触发单层转动，并保证动画完成后 3D 视觉状态与 `CubeState` 逻辑状态一致。

该 milestone 的增量价值是让页面从“可看、可切换”进入“可手动操作”，同时验证输入识别、动作公共契约、动画事务和状态提交链路。

## 范围

包含：

- 右键视角控制：
  - 用户在魔方区域右键拖动时，视角随拖动变化。
  - 右键拖动不改变 `CubeState`。
  - 右键拖动不改变打乱步骤数和求解步骤数。
  - 右键拖动过程中避免浏览器默认右键菜单干扰主要交互。
- 左键层转动交互：
  - 用户在魔方上左键按下并拖动。
  - 系统基于命中信息、拖动方向和当前视角生成候选 `InteractionIntent` 或候选 `Move`。
  - 拖动距离不足或方向无法判断时，不触发层转动。
  - 有效拖动触发单层旋转动画。
  - 动画完成后通过魔方逻辑状态模型提交对应 `Move`。
- 单步动画事务：
  - 单次手动层转动期间阻止冲突操作。
  - 动画完成后渲染结果与新的 `CubeState` 对齐。
  - 动画失败时不得让视觉状态和逻辑状态长期不一致。
- 计数隔离：
  - 手动层转动不计入打乱步骤数。
  - 手动层转动不计入求解步骤数。
- 基础反馈：
  - 对无效拖动、动画中暂不可操作等状态提供可理解反馈或至少不造成误触发。

不包含：

- 不实现自动随机打乱。
- 不实现自动求解。
- 不实现复杂手势预测、触屏交互或键盘替代操作。
- 不实现教学模式、公式提示或逐步讲解。
- 不改变 3/4/5 阶范围。
- 不承诺任何求解器能力。
- 不要求所有可能鼠标拖动都能被自然识别，只要求形成可验收的基本判定规则。

风险与依赖：

- 左键拖动识别依赖 3D 命中测试、相机视角和层规则，容易与渲染细节耦合；必须保持输入模块只产出意图，不直接修改 `CubeState`。
- 不同阶数的可转动层规则必须基于统一 `Move` 语义，不能硬编码只适配 3 阶后再返工。
- 若动画提交策略在实现中确定为“动画完成后提交”或“逻辑先提交再视觉对齐”，必须保持事务可测，并避免视觉/逻辑不一致。

## 预计 Task 切片

1. 右键视角控制  
   实现右键拖动相机/视角变化，屏蔽默认右键菜单干扰，并验证不影响魔方逻辑状态和计数。

2. 渲染命中测试与输入临时状态  
   建立鼠标命中魔方面/小块的公开能力，以及左键拖动起点、方向、距离等临时状态管理。

3. 左键拖动到 `InteractionIntent` / `Move` 的判定  
   实现最小有效拖动阈值、方向判定、候选层/方向生成，并覆盖无效拖动不触发。

4. 手动单步动画事务  
   将合法 `Move` 交给动画调度与渲染层播放，动画完成后提交 `CubeState`，并处理动画中禁用冲突操作。

5. 手动交互状态一致性与计数隔离测试  
   覆盖右键不改状态、左键有效转动改状态、无效拖动不改状态、手动转动不影响两个自动计数器。

## 验收

- 用户在魔方区域右键拖动时，视角发生可见变化。
- 右键拖动后：
  - `CubeState` 不发生层转动变化。
  - 打乱步骤数不增加。
  - 求解步骤数不增加。
  - 浏览器默认右键菜单不应导致主要交互不可用。
- 用户在魔方上左键拖动并形成有效方向时，应触发某一层转动。
- 层转动有可见变化，且动画完成后视觉状态与逻辑状态一致。
- 一次有效手动层转动后，魔方状态不同于转动前状态，除非被后续反向操作抵消。
- 拖动距离不足或方向无法判断时，不误触发层转动。
- 手动层转动不计入自动打乱步骤数。
- 手动层转动不计入求解步骤数。
- 自动动画或手动单步事务执行中，不允许插入冲突层转动导致状态错乱。
- 输入交互模块不直接改写 `CubeState`，必须经由应用控制层、动作校验和动画事务提交。
- 相关输入判定、状态提交和计数隔离具备可测试证据。
