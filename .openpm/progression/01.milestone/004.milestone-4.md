# 004.milestone-4

- 来源：DESIGN.md + MSC.md

## 目标

完成 MVP 版本验收闭环：在已有 3D 展示、阶数切换、手动交互和随机打乱基础上，交付自动求解、求解步骤计数、求解失败反馈和整体回归验收，使用户能够完成“打开页面 → 操作魔方 → 随机打乱 → 自动求解复原 → 查看步骤数”的核心路径。

该 milestone 是 MVP 版本验收 milestone，必须覆盖 DESIGN 中 MVP 可交付闭环，但不在“最优解”定义未确认前承诺数学意义全局最优。

## 范围

包含：

- 求解器 / 求解适配器公共边界：
  - 接收 `SolverRequest`，包含当前 `CubeSize`、`CubeState` 快照、可用 move set 和能力要求。
  - 返回 `SolverResult`，能够表达成功、已完成、失败、不支持、动作序列、步骤数、最优性状态和原因。
  - 已完成态点击求解返回空动作序列或等价成功结果，步骤数为 0。
- 自动求解流程：
  - 用户点击“求解”后，系统基于当前 `CubeState` 生成求解步骤。
  - 成功时，将求解序列交给自动播放模块逐步执行。
  - 每步动画完成后提交对应 `Move`，保持视觉与逻辑一致。
  - 全部完成后魔方恢复完成态。
- 求解步骤计数：
  - 固定一种可验证展示口径：
    - 来自 `SolverResult` 动作序列长度；或
    - 来自自动播放实际成功执行的求解动作数。
  - 求解完成后显示最终求解步骤数。
  - 手动转动不直接计入求解步骤数。
- 求解失败与不支持反馈：
  - 求解失败时向用户展示可理解反馈。
  - 求解失败时不启动错误动画队列。
  - 求解失败时不得让魔方进入视觉状态和逻辑状态不一致。
  - 对暂不支持的阶数或状态给出明确反馈，而不是静默失败。
- 自动求解互斥：
  - 求解期间禁止重复求解。
  - 求解期间禁止随机打乱。
  - 求解期间禁止手动层转动。
  - 求解期间禁止阶数切换，或明确禁用并反馈。
- MVP 回归验收：
  - 页面初始状态。
  - 3/4/5 阶切换。
  - 右键视角控制。
  - 左键手动层转动。
  - 随机打乱 20 步。
  - 自动求解复原。
  - 打乱步骤数和求解步骤数。
  - 速度滑块对自动打乱和自动求解均生效。
  - 自动流程互斥。
  - 错误反馈与基础可访问文本标签。

不包含：

- 不承诺“最优解”已满足，除非用户已明确补充最优解度量、适用阶数、耗时上限和验收方式。
- 不要求 3/4/5 阶都达到数学全局最短解。
- 不实现教学模式、公式说明、逐步提示。
- 不实现账号、云存储、历史记录、排行榜、分享。
- 不实现移动端触摸手势。
- 不实现打乱/求解中断、取消、重启或队列能力。
- 不引入远程日志、监控、分析统计或用户数据收集。
- 不扩展 3/4/5 阶以外的魔方类型。

风险与依赖：

- “最优解”定义仍是最大产品风险；MVP 只能验收“自动求解复原”和 `SolverResult` 正确表达最优性能力状态，不能把非最优复原误标为最优。
- 4 阶、5 阶求解能力可能受算法复杂度影响；若实施阶段无法在浏览器内稳定支持，应通过 `SolverResult` 明确不支持或失败原因，并由 Producer/Architect/用户确认是否接受 MVP 降级。
- 如引入求解库、Web Worker 或复杂异步求解策略，需说明依赖、性能、兼容性和维护风险，并进入 ADR 候选或架构复核。
- 自动求解失败不得破坏当前魔方状态；失败恢复和错误反馈是验收关键。

## 预计 Task 切片

1. `SolverRequest` / `SolverResult` 契约与求解能力边界  
   建立求解器输入输出公共契约，明确成功、已完成、失败、不支持、步骤数和最优性状态表达。

2. 最小可复原求解策略或适配器接入  
   实现或接入能够基于当前 `CubeState` 返回可执行求解序列的能力，并保持可替换边界；如仅支持部分阶数，必须明确不支持反馈。

3. 已完成态与求解失败反馈  
   覆盖已完成态求解步骤数为 0 或等价反馈，以及求解失败/不支持时的 UI 提示和状态一致性。

4. 自动求解播放与状态提交  
   将求解序列接入自动播放模块，逐步执行动画、提交 `CubeState`、更新求解进度，并在完成后验证完成态。

5. 求解步骤计数与速度控制  
   固定求解步骤数展示口径，并让速度滑块影响自动求解播放速度。

6. 求解流程互斥与控件状态  
   求解期间禁用重复求解、打乱、手动转动和阶数切换，冲突操作提供反馈。

7. MVP 端到端 smoke 与回归验收  
   覆盖打开页面、阶数切换、右键视角、左键转动、随机打乱 20 步、自动求解复原、步骤计数、速度控制、错误反馈和互斥规则。

8. MVP 风险确认与交接  
   汇总最优解未确认、求解能力边界、性能风险、第三方依赖风险和后续需要用户/Architect/QA 确认的事项。

## 验收

- 页面初始状态满足：
  - 显示 3D 魔方主体。
  - 初始魔方处于完成态。
  - 显示随机打乱按钮、求解按钮、两个步骤计数器、速度滚动条和 3/4/5 阶切换入口。
  - 初始打乱步骤数和求解步骤数为 0 或等价明确空状态。
- 阶数切换满足：
  - 选择 3 阶显示 3 阶魔方。
  - 选择 4 阶显示 4 阶魔方。
  - 选择 5 阶显示 5 阶魔方。
  - 切换后魔方处于对应阶数完成态。
  - 切换后两个计数器重置为 0。
  - 切换后不继续播放上一阶动画或保留上一阶内部状态。
- 右键视角控制满足：
  - 右键拖动改变视角。
  - 不改变 `CubeState`。
  - 不增加打乱步骤数。
  - 不增加求解步骤数。
- 左键手动层转动满足:
  - 有效拖动触发层转动。
  - 无效拖动不误触发。
  - 动画完成后视觉状态与逻辑状态一致。
  - 手动转动不计入打乱或求解步骤数。
- 随机打乱满足：
  - 点击随机打乱后自动执行 20 步。
  - 每一步更新视觉状态和逻辑状态。
  - 完成后打乱步骤数为 20。
  - 完成后魔方处于可感知打乱态。
  - 打乱期间不允许重复打乱、求解、手动层转动或阶数切换导致状态重叠。
- 自动求解满足：
  - 点击求解后，系统基于当前魔方状态生成求解步骤或明确失败/不支持原因。
  - 求解步骤执行过程中，每一步更新视觉状态和逻辑状态。
  - 求解完成后魔方恢复完成态。
  - 求解完成后求解步骤数显示本次求解所用步数。
  - 如果魔方已经是完成态，点击求解后保持完成态，求解步骤数显示 0 或等价反馈。
  - 求解失败时显示可理解错误反馈，不静默失败。
  - 求解失败时不进入视觉状态和逻辑状态不一致。
  - 求解期间不允许重复求解、随机打乱、手动层转动或阶数切换导致状态重叠。
- 速度控制满足：
  - 速度滑块调整后，自动打乱速度发生可感知变化。
  - 速度滑块调整后，自动求解速度发生可感知变化。
  - 最小值与最大值下自动流程仍可完成或至少保持状态正确。
  - 自动流程中调整速度不破坏状态一致性。
- 可访问性与反馈满足：
  - 关键按钮和控件有可理解文本标签。
  - 步骤计数器使用文本展示，不能仅依赖颜色或动画。
  - 求解失败、自动流程进行中、无效拖动、渲染能力不足等场景有可理解反馈或可诊断失败路径。
- 架构边界满足：
  - `CubeState` 仍由魔方逻辑状态模型唯一拥有。
  - 渲染层只派生视觉状态。
  - 输入模块不直接改写 `CubeState`。
  - 求解器不直接修改 `CubeState`，不直接驱动 3D 渲染。
  - 自动播放模块持有动作队列和执行进度。
  - 速度滑块只影响播放速度，不改变动作序列或逻辑状态。
- “最优解”验收限制：
  - 在用户确认最优解定义、适用阶数、计数度量和耗时上限前，不得宣称 MVP 已满足数学意义最优解。
  - `SolverResult` 必须能表达是否保证最优、非最优、失败或不支持。
- MVP 通过条件：
  - 用户能够完成“打开页面 → 查看完成态魔方 → 切换 3/4/5 阶 → 右键调整视角 → 左键手动转动 → 随机打乱 20 步 → 自动求解恢复完成态 → 查看打乱与求解步骤数 → 调整自动速度”的完整闭环。
  - 上述闭环有 smoke 或端到端验收证据。
  - 已知降级、未确认最优性和依赖风险被明确记录并交接给后续 Producer / Architect / QA。
