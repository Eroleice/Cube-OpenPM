# 003.milestone-3

- 来源：DESIGN.md + MSC.md

## 目标

交付随机打乱与自动播放闭环：用户点击“随机打乱魔方”后，系统自动执行 20 步合法随机转动，按速度滑块控制播放速度，更新打乱步骤数，并在流程期间保持互斥，防止重复打乱、求解、手动转动和阶数切换导致状态混乱。

该 milestone 的核心增量是验证自动流程状态机、动作队列、动画调度、步骤计数和速度参数共同工作。

## 范围

包含：

- 随机打乱序列生成：
  - 基于当前 `CubeSize` 生成 20 步合法 `MoveSequence`。
  - 避免明显无效或等价抵消的序列，使打乱结果可感知。
  - 序列标记为 `scramble`，用于计数与流程状态。
- 自动播放与动画调度：
  - 接收 `MoveSequence` 后逐步播放层转动动画。
  - 每步动画完成后提交对应 `Move` 到 `CubeState`。
  - 每步完成后更新打乱步骤数。
  - 第 20 步完成后回到 `idle`。
- 打乱计数：
  - 打乱开始时重置本次打乱计数。
  - 自动打乱完成后打乱步骤数显示 20。
  - 手动转动仍不影响打乱步骤数。
- 速度控制：
  - 速度滑块具备最小值、最大值和默认值。
  - 速度影响自动打乱每步动画速度或播放间隔。
  - 自动流程中调整速度时，后续步骤尽量按新速度生效，不破坏状态一致性。
- 自动流程互斥：
  - 打乱中禁止重复打乱。
  - 打乱中禁止求解。
  - 打乱中禁止手动层转动。
  - 打乱中禁止阶数切换，或至少明确禁用并反馈。
- 基础错误与反馈：
  - 非 `idle` 状态触发冲突操作时，展示“自动流程进行中”等可理解反馈。
  - 自动播放失败时停止流程并保持或恢复一致状态。

不包含：

- 不实现自动求解能力。
- 不实现打乱中断、取消、重启或队列能力。
- 不实现求解失败细节反馈。
- 不承诺随机打乱达到任何数学随机性或难度分布标准。
- 不新增存档、历史记录、计时比赛或分享能力。

风险与依赖：

- 自动流程互斥是 MVP 稳定性的关键，不能为了体验扩展队列、中断或重启；如后续要求取消/重启，需进入 ADR 或需求变更。
- 速度滑块只影响播放速度，不得改变 `MoveSequence`、求解结果或 `CubeState`。
- 20 步打乱后的非完成态需通过生成规则或结果校验保障；极端回到完成态时应重新生成或避免等价无效序列。

## 预计 Task 切片

1. 自动播放队列与 `RunMode` 互斥  
   建立自动播放队列、执行进度、`scrambling` 状态，以及流程期间按钮/阶数/手动转动禁用规则。

2. 随机打乱 20 步序列生成  
   基于 3/4/5 阶合法 move set 生成固定 20 步 `MoveSequence`，避免明显抵消和完成态结果。

3. 打乱逐步动画与状态提交  
   将 20 步队列逐步转化为动画命令，动画完成后提交 `CubeState` 并更新 3D 视觉。

4. 打乱步骤计数闭环  
   实现打乱计数从执行进度派生，开始重置、逐步递增、完成为 20，并覆盖手动转动不计数。

5. 速度滑块控制自动播放  
   定义速度最小值、最大值、默认值，接入自动打乱播放节奏，并验证运行中调整速度不破坏流程。

6. 自动流程冲突反馈与失败恢复  
   对打乱中重复点击、求解点击、阶数切换和手动转动提供禁用或反馈，并保证失败时状态一致。

## 验收

- 点击“随机打乱魔方”后，系统自动执行 20 步随机合法转动。
- 自动打乱过程中，每一步都更新魔方视觉状态和逻辑状态。
- 自动打乱完成后，打乱步骤数显示 20。
- 自动打乱完成后，魔方处于非完成态；若随机序列可能回到完成态，实现应避免生成等价无效打乱或重新生成。
- 自动打乱期间不允许重复启动另一个打乱流程导致状态重叠。
- 自动打乱期间不允许同时启动求解流程导致状态重叠。
- 自动打乱期间不允许手动层转动破坏动作队列。
- 自动打乱期间阶数切换被禁用，或提供明确不可操作反馈；不得继续播放到错误阶数。
- 速度滑块调整后，自动打乱速度发生可感知变化。
- 速度滑块处于最小值时，自动转动仍能完成，不应卡死。
- 速度滑块处于最大值时，自动转动仍至少保持状态正确，不应跳步导致状态错误。
- 自动流程中调整速度不破坏当前自动流程的状态一致性。
- 打乱步骤数由自动播放实际执行进度驱动，而不是 UI 自行推断。
- 相关自动流程、计数、互斥和速度控制具备测试或 smoke 证据。
