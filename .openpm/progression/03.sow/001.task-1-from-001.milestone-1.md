# 001.task-1-from-001.milestone-1

- 来源 Task: `.openpm/progression/02.task/001.task-1-from-001.milestone-1.md`
- 来源 Milestone: `.openpm/progression/01.milestone/001.milestone-1.md`

## 施工目标

建立浏览器页面入口与基础 UI 初始状态，让用户打开页面后能看到 3D 魔方程序的最小界面骨架。

本次施工只交付当前 task 要求的用户可见初始体验：

1. 页面存在一个可识别的魔方展示/承载区域。
2. 页面显示“随机打乱”按钮。
3. 页面显示“求解”按钮。
4. 页面显示“速度”或等价含义的自动转动速度滚动条。
5. 页面显示“3 阶”“4 阶”“5 阶”切换入口。
6. 页面显示打乱步骤数计数器，初始为 `0` 或等价明确空状态。
7. 页面显示求解步骤数计数器，初始为 `0` 或等价明确空状态。
8. 默认 UI 表达为 3 阶选中，或有等价的默认 3 阶初始状态。
9. “随机打乱”“求解”“速度”在本 task 中只能作为基础控件或占位入口展示，必须避免误导用户认为随机打乱、自动求解或自动播放已经完成。
10. 增加最小 smoke 覆盖，证明页面初始 UI、基础控件、计数器和魔方承载区域可见。

本 task 不要求真实 3D 魔方渲染完成，允许魔方区域使用明确占位容器，但该占位不得固化与后续 `CubeSize`、`CubeState` 冲突的业务状态结构。

## 当前事实

1. 产品目标来自 `DESIGN.md`：
   - 浏览器中运行的 3D 魔方交互程序。
   - 最终需要支持 3 阶、4 阶、5 阶。
   - 页面需要展示随机打乱按钮、求解按钮、打乱步骤数、求解步骤数、速度滚动条、阶数切换入口。
   - 首次进入页面默认建议为 3 阶，打乱步骤数和求解步骤数初始建议为 `0`。

2. 架构边界来自 `MSC.md`：
   - UI 控制与状态展示模块负责展示页面控件、计数、运行状态和反馈。
   - UI 不直接修改 `CubeState`。
   - `CubeState` 的唯一权威 owner 是魔方逻辑状态模型模块。
   - 3D 渲染层只消费逻辑状态并派生视觉表现，不作为业务状态源。
   - 当前技术栈、前端框架、3D 库、求解算法未由 DESIGN 或 MSC 指定。
   - 引入第三方 3D 库、求解库、公共契约或重大架构决策需要后续架构复核，不在本 task 内预批准。

3. 当前 task 明确约束：
   - 聚焦页面入口、基础布局、初始控件展示、初始计数器文本和最小 smoke 覆盖。
   - 不实现真实 3D 完成态魔方渲染。
   - 不实现 `CubeState`、`Move`、`MoveSequence`、`RunMode` 或 `PlaybackState` 公共契约。
   - 不实现 3/4/5 阶切换后的状态重建闭环。
   - 不实现随机打乱、自动求解、鼠标交互或自动播放。
   - UI 临时状态只能用于页面展示，不得成为 `CubeState` 的替代权威。

4. 执行前必须由 OpenCode 识别真实仓库事实，不得假设项目一定使用 React、Vite、Three.js、Vitest、Testing Library、Playwright 或任何特定框架。至少记录：
   - 包管理器。
   - 构建脚本。
   - 测试脚本。
   - 前端入口。
   - 源码目录结构。
   - 测试框架。
   - 是否存在 `tsconfig.json`。
   - 如存在 TypeScript 配置，当前 strict mode 状态。

5. 如果项目包含 `tsconfig.json`：
   - 必须保持 TypeScript strict mode。
   - 不得关闭 strict。
   - 不得跳过类型检查。
   - 不得通过放宽 `tsconfig` 完成任务。

6. 如仓库无测试能力且建立 smoke 需要新增测试框架、复杂配置或依赖，必须停止并报告，不能自行引入。
7. 测试不应依赖真实打乱、求解、自动播放、3D 渲染或领域状态模型。

## 验收标准

以下验收标准逐条追溯当前 task 的“独立验收标准”：

1. 魔方承载区域
   - 验收项：打开页面后能看到一个用于承载 3D 魔方主体的可识别区域。
   - 证据要求：smoke 测试断言或人工验证说明中能定位该区域。
   - 允许实现：明确占位容器。
   - 禁止实现：伪装为真实 3D 渲染完成，或引入未经复核 3D 库。

2. “随机打乱”按钮
   - 验收项：页面显示“随机打乱”按钮。
   - 证据要求：smoke 测试通过文本/role/accessibility name 查询到按钮。
   - 额外要求：按钮不得让用户误以为随机打乱功能已完成；需 disabled、说明文案或可见提示。

3. “求解”按钮
   - 验收项：页面显示“求解”按钮。
   - 证据要求：smoke 测试通过文本/role/accessibility name 查询到按钮。
   - 额外要求：按钮不得让用户误以为自动求解功能已完成；需 disabled、说明文案或可见提示。

4. “速度”控件
   - 验收项：页面显示“速度”或等价含义的自动转动速度滚动条。
   - 证据要求：smoke 测试能识别 slider/range 或等价控件，并能看到“速度”标签或 accessible name。
   - 额外要求：当前只能作为入口展示，不得暗示自动播放速度控制已经完成。

5. 阶数切换入口
   - 验收项：页面显示“3 阶”“4 阶”“5 阶”入口。
   - 证据要求：smoke 测试或人工验证能分别识别三个入口。
   - 额外要求：本 task 只要求初始展示，不实现切换后重建 `CubeState` 或更新真实 3D 魔方。

6. 默认 3 阶表达
   - 验收项：默认 UI 状态表达为 3 阶选中或等价默认 3 阶初始状态。
   - 证据要求：smoke 测试或人工验证能证明默认 3 阶可识别，例如选中态、说明文案、`aria-pressed`、默认 label 或等价表达。
   - 额外要求：该状态仅为 presentation state，不得成为 `CubeSize` 公共领域契约。

7. 打乱步骤数计数器
   - 验收项：页面显示打乱步骤数计数器，初始为 `0` 或等价明确空状态。
   - 证据要求：smoke 测试或人工验证能看到“打乱步骤数”与 `0` 或明确空状态。
   - 额外要求：不得实现真实打乱计数逻辑。

8. 求解步骤数计数器
   - 验收项：页面显示求解步骤数计数器，初始为 `0` 或等价明确空状态。
   - 证据要求：smoke 测试或人工验证能看到“求解步骤数”与 `0` 或明确空状态。
   - 额外要求：不得实现真实求解计数逻辑。

9. 未完成功能不误导
   - 验收项：“随机打乱”“求解”“速度”仅作为基础控件展示，不误导用户认为随机打乱、自动求解或自动播放已完成。
   - 证据要求：测试或人工验证能看到 disabled、占位提示、说明文案，或点击后有明确提示。
   - 禁止：静默 no-op。

10. smoke 覆盖
   - 验收项：用户可见初始页面具备 smoke 覆盖，至少证明基础控件、计数器和魔方承载区域可见。
   - 证据要求：coding report 中列出 smoke 测试文件、执行命令和结果。
   - 若无法自动化覆盖：必须说明真实阻塞原因、停止条件是否触发、替代人工验证步骤和后续风险；不能无理由跳过。

11. 架构边界
   - 验收项：本 task 不引入 UI、输入模块或渲染模块直接改写魔方逻辑状态的设计；临时 UI 状态只用于页面展示。
   - 证据要求：代码 diff 中不得出现 `CubeState`、`Move`、`MoveSequence`、`RunMode`、`PlaybackState` 等领域 contract 的实现或公共导出。
   - 额外要求：不得新增公共 API、schema、事件协议、持久化格式或跨模块共享状态。

## 风险

1. 仓库事实未知风险
   - 当前 SOW 生成时未直接确认真实源码结构、构建入口和测试框架。
   - 缓解：OpenCode 必须先识别仓库事实；若缺少前端入口、构建脚本或测试能力且需要重大工程结构，必须停止并报告。

2. 范围滑坡风险
   - 基础 UI 容易被扩展为真实 3D 渲染、领域模型、阶数切换、打乱、求解、动画或鼠标交互。
   - 缓解：本 SOW 已明确禁止越界内容；audit 需检查 diff 是否只覆盖页面入口、基础 UI、轻量样式和 smoke。

3. 用户误导风险
   - 如果按钮可点击但无反馈，用户可能误以为随机打乱或求解已实现。
   - 缓解：必须使用 disabled、占位提示、说明文案或带明确提示的 no-op；静默 no-op 不可接受。

4. UI 临时状态变成领域权威的风险
   - 默认 3 阶、计数器 `0`、速度初始值等 presentation state 可能被后续误用为 `CubeState` 或 `CubeSize` 权威来源。
   - 缓解：本 task 不新增领域 contract；状态仅限页面展示，后续 task 必须以领域状态模块为权威。

5. smoke 覆盖不足风险
   - task 明确要求 smoke 覆盖；若只做人工查看，后续 QA audit 可能无法放行。
   - 缓解：优先沿用仓库已有测试框架；如建立 smoke 需要新增框架或复杂配置，停止并报告，由 Producer/Architect 决定是否拆分工程入口或测试基础任务。

6. TypeScript strict 回归风险
   - 如果项目为 TypeScript，关闭 strict 或跳过类型检查会违反全局规则。
   - 缓解：不得修改 `tsconfig` 放宽约束；必须执行可用类型检查或记录真实缺口。

7. 依赖风险
   - 本 task 不需要第三方 3D 库、UI 框架、状态管理库或新测试框架。
   - 缓解：不得新增未经复核依赖；如判断必须新增，停止并交回 Producer/Architect 复核，必要时补 ADR。

8. 可访问性与可测试性风险
   - 如果控件缺少文本、label 或 accessible name，smoke 查询和基础可访问性都会变差。
   - 缓解：关键控件必须使用可理解文本标签或 accessible name，包括“随机打乱”“求解”“速度”“3 阶”“4 阶”“5 阶”；计数器使用文本展示，不仅依赖颜色或图标。
