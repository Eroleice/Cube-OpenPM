# 000.project_baseline_setup

## 施工目标

- 建立或修复当前项目的最小工程基线，使后续业务 SOW 可以基于真实仓库事实执行。
- 完成本 SOW 后，OpenPM 将回到原业务 task 继续生成对应 SOW。

## 当前事实

- package.json：不存在
- 包管理器：未确认
- scripts：无
- dependencies：无
- devDependencies：无
- 入口文件：未发现
- 源码目录：未发现
- 测试目录：未发现
- 配置文件：未发现
- 测试框架：未发现
- 缺失或冲突：未发现 package.json，无法确认包管理器、脚本和依赖基线。；未发现 lockfile 或 packageManager 字段，无法确认包管理器。；未发现可确认的应用入口文件。；未发现 src/ 或 app/ 源码目录。；package.json 缺少 build 脚本。；package.json 缺少 test 脚本。；未发现明确测试框架或测试配置。

## 修改范围

- 可以新增或修复 package.json、lockfile、tsconfig、Vite/Vitest/Jest 配置、入口文件、src/ 源码目录和 tests/ 测试目录。
- 可以新增最小 smoke 测试，用于证明测试框架可运行。
- 包管理器按当前事实使用：待用户确认，建议默认 npm。
- ADR owner 固定为：Architect；OpenCode 更新或引用工程基线 ADR 时必须直接填写该 owner，不得使用 OpenCode 作为长期 owner。

## 建议修改代码文件

- package.json
- tsconfig.json
- vite.config.ts 或现有等价构建配置
- vitest.config.ts 或现有等价测试配置
- index.html
- src/main.tsx 或项目实际入口

## 建议新增代码文件

- src/main.tsx
- src/App.tsx
- tests/app.smoke.test.ts

## 不做

- 不实现原业务 task 的 UI、交互或业务验收项。
- 不引入与工程基线无关的依赖。
- 不修改 001 及后续业务 task 的内容。

## 必跑检查

- npm run build
- npm run typecheck
- npm run test

## 测试要求

- 至少新增或修复一个 smoke 测试，证明测试框架、DOM 环境和入口模块可以被执行。
- 如果项目不是前端项目，必须在 Coding Report 中说明等价入口、测试框架和检查命令。

## 验收标准

- package.json 能明确 scripts、依赖和包管理器事实。
- 应用入口文件、源码目录和测试目录可被后续 SOW 引用。
- build/typecheck/test 命令存在；不适用的命令必须给出项目事实依据。
- 本 SOW 的 Coding Report 能说明所有新增文件、脚本、依赖和命令结果。

## Coding Report 证据要求

- 列出 package.json 当前状态、脚本、依赖和变更说明。
- 说明新增依赖或脚本的必要性、替代方案和维护风险。
- 列出入口文件、源码目录、测试目录和测试框架证据。
- 列出 smoke 测试文件与断言范围。
- 列出 build、typecheck、test 命令与结果；失败、跳过或不适用必须说明原因。
- 更新或引用 .openpm/progression/00.documents/ADR.md 中的工程基线决策，列出 ADR 编号、固定 owner（Architect）、包管理器、兼容策略、迁移或回滚影响；如果不需要新增 ADR，必须说明理由。
- 如果 package.json 写入或修改 packageManager，区分项目声明/期望版本和本地执行环境版本；版本不一致但未影响功能实现、依赖稳定性和必跑检查时不单独阻塞，若导致安装、构建、测试或 SOW 验收失败则必须作为版本冲突阻塞并要求 rework 对齐版本。
- 说明 git status 中未跟踪文件、OpenPM progression artifact 和产品/测试/配置文件的处理方式。

  - 回复：是

  - 回复：是

## 风险

- 工程基线会新增或调整项目结构；必须由用户确认后才能进入 code。
- 如果用户希望使用不同技术栈或包管理器，应先在待确认事项中明确回复。
