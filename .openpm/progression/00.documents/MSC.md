# MSC

## 架构目标

本架构面向一个运行于现代浏览器中的 3D 魔方交互程序，目标是把 `DESIGN.md` 中的用户可见行为转化为稳定的模块边界、状态归属、通信方向和公共内部契约。

核心架构目标如下：

1. 支撑浏览器内 3D 魔方展示，覆盖 3 阶、4 阶、5 阶魔方。
2. 保证魔方逻辑状态、3D 视觉表现、动画过程、步骤计数和按钮状态一致。
3. 隔离两类鼠标交互：
   - 右键拖动只控制观察视角，不改变魔方逻辑状态，不计入任何步骤数。
   - 左键拖动只在形成有效层转动后生成转动动作，并通过统一动作执行链路更新状态。
4. 统一表达手动转动、随机打乱、自动求解所使用的魔方转动动作，避免各模块各自定义动作语义。
5. 支撑随机打乱固定执行 20 步，并由自动执行进度驱动打乱步骤计数。
6. 支撑求解器按当前魔方状态返回可执行动作序列，并由求解结果或实际执行进度驱动求解步骤计数。
7. 将“最优解”作为待确认强需求候选，不在架构中承诺具体算法、复杂度、最大求解时间或 3/4/5 阶均可数学最优。
8. 默认采用保守自动流程互斥策略，避免打乱、求解、手动转动、阶数切换并发导致状态错乱。
9. 将速度滚动条建模为自动播放参数，只影响动画速度或播放间隔，不改变动作序列、求解结果或魔方逻辑状态。
10. 保持模块可替换和可测试，尤其是魔方状态模型、3D 渲染、输入识别、动画调度和求解器能力。

## 模块列表

### 1. UI 控制与状态展示模块

职责：

- 展示页面控件：
  - 随机打乱按钮。
  - 求解按钮。
  - 打乱步骤数计数器。
  - 求解步骤数计数器。
  - 自动转动速度滚动条。
  - 3 阶、4 阶、5 阶切换入口。
  - 错误或不支持提示。
- 将用户点击、选择和滑动操作转交给应用控制层。
- 根据流程状态禁用或启用按钮与阶数切换控件。
- 展示来自流程状态模块的计数、运行状态和错误反馈。

不负责：

- 不直接修改 `CubeState`。
- 不自行推断打乱步骤数或求解步骤数。
- 不直接驱动 3D 动画。
- 不实现求解算法。

状态 owner：

- UI 展示状态由本模块消费，但计数和运行状态的权威来源是流程状态与自动播放模块。
- 速度滑块当前值可由应用状态持有，UI 负责展示和输入。

### 2. 应用控制层 / 流程协调模块

职责：

- 接收 UI 事件和输入交互模块产出的意图。
- 根据当前 `RunMode` 判断操作是否允许。
- 编排以下用户路径：
  - 初始化页面。
  - 切换 3/4/5 阶。
  - 手动层转动。
  - 随机打乱。
  - 自动求解。
  - 调整速度。
- 负责模块之间的调用顺序，保证单向数据流。
- 将错误、失败、不支持状态转化为可展示反馈。

不负责：

- 不保存魔方贴纸或小块排列的权威状态。
- 不直接计算求解算法。
- 不直接操作 3D 对象内部结构。

状态 owner：

- 当前阶数 `CubeSize` 建议由应用控制层或应用状态容器统一持有。
- 当前运行模式 `RunMode` 由应用控制层与自动播放模块共同约束，但权威更新应集中在流程协调边界内。

### 3. 魔方逻辑状态模型模块

职责：

- 拥有并维护 `CubeState` 的权威逻辑状态。
- 根据 `CubeSize` 创建对应阶数的完成态。
- 校验 `Move` 对当前阶数是否合法。
- 应用合法 `Move` 并产出新的确定性逻辑状态。
- 判断当前魔方是否处于完成态。
- 为求解器提供当前状态快照。

不负责：

- 不处理鼠标事件。
- 不处理 3D 相机或材质动画。
- 不负责按钮禁用。
- 不负责自动播放节奏。
- 不自行决定随机打乱序列或求解序列。

状态 owner：

- `CubeState` 的唯一权威 owner。
- 动画中的临时视觉状态不得覆盖 `CubeState`。
- 自动动画完成后，必须通过本模块提交对应 `Move` 的逻辑状态变化。

### 4. 3D 渲染与场景模块

职责：

- 根据当前 `CubeSize` 和 `CubeState` 构建或更新魔方视觉表现。
- 表达每个小块或每个面的颜色状态。
- 展示完成态、打乱态、求解中间态。
- 管理 3D 场景、相机、灯光、材质、几何体等渲染细节。
- 执行层旋转动画的视觉表现。
- 暴露必要的命中测试能力，供输入交互模块识别鼠标命中的魔方面或小块。

不负责：

- 不作为 `CubeState` 的权威状态源。
- 不自行修改魔方逻辑状态。
- 不决定动作是否合法。
- 不生成随机打乱序列。
- 不生成求解序列。
- 不维护步骤计数。

状态 owner：

- 视觉状态由 `CubeState` 与当前动画进度派生。
- 相机/视角状态由渲染模块或输入交互模块下的视角控制子模块持有，但该状态不属于魔方逻辑状态。

### 5. 输入交互识别模块

职责：

- 处理鼠标右键拖动视角控制。
- 处理鼠标左键拖动魔方层转动识别。
- 使用命中测试结果、拖动起点、拖动方向、当前视角等信息生成交互意图。
- 对左键拖动执行基本有效性判断：
  - 拖动距离不足时不生成层转动。
  - 方向无法明确判断时不生成层转动。
  - 有效时生成候选 `InteractionIntent` 或候选 `Move`。

不负责：

- 不直接修改 `CubeState`。
- 不直接更新步骤计数。
- 不直接进入求解器。
- 不绕过应用控制层执行动画。

状态 owner：

- 手动拖动临时状态由本模块持有，仅在一次交互期间有效。
- 视角拖动状态与魔方逻辑状态隔离。

### 6. 动作与序列模块

职责：

- 定义统一的 `Move`、`MoveSequence` 和动作计数语义。
- 为手动转动、随机打乱、自动求解提供同一动作表达。
- 提供动作合法性辅助、反向动作、序列规范化等可测试规则。
- 将逻辑动作转化为动画调度可消费的 `AnimationCommand` 或等价协议。

不负责：

- 不拥有当前 `CubeState`。
- 不直接播放动画。
- 不决定 UI 是否禁用。
- 不实现求解算法。

状态 owner：

- 不持有长期业务状态。
- `MoveSequence` 可作为流程中的一次性队列数据，由自动播放模块持有执行进度。

### 7. 自动播放与动画调度模块

职责：

- 接收 `MoveSequence` 并按当前速度参数逐步执行。
- 将每个 `Move` 转化为动画命令并调用 3D 渲染模块播放。
- 在每步动画完成后通知魔方逻辑状态模型提交对应 `Move`。
- 维护当前动作队列、执行进度和播放状态。
- 为打乱步骤计数和求解步骤计数提供执行进度依据。
- 保证同一时间只运行一个自动流程或一个受控动画事务。

不负责：

- 不生成随机动作序列。
- 不生成求解动作序列。
- 不自行决定魔方状态变更规则。
- 不让 UI 直接插入未校验动作。

状态 owner：

- 动作队列和当前执行进度由本模块持有。
- `PlaybackState` / `RunMode` 中与播放有关的状态由本模块与应用控制层协作维护。
- 动画中间态是临时表现状态，不是权威 `CubeState`。

### 8. 随机打乱模块

职责：

- 基于当前 `CubeSize` 生成 20 步随机合法 `MoveSequence`。
- 避免明显无效或等价抵消的打乱序列，使用户可感知魔方被打乱。
- 将生成的打乱序列交给自动播放模块执行。
- 标记该序列为 `scramble` 用于计数和流程状态。

不负责：

- 不直接播放动画。
- 不直接修改 `CubeState`。
- 不维护按钮状态。
- 不执行求解。

状态 owner：

- 不持有长期状态。
- 打乱步骤数来自自动播放模块执行该 20 步队列的进度，完成后应为 20。

### 9. 求解器 / 求解适配器模块

职责：

- 接收 `SolverRequest`，包含当前 `CubeSize`、`CubeState`、可用 move set、最优性要求或能力要求。
- 返回 `SolverResult`，包含：
  - 是否成功。
  - 动作序列。
  - 步骤数。
  - 是否保证最优或最优性状态。
  - 失败原因。
  - 不支持原因。
  - 能力边界说明。
- 对已完成态返回空动作序列或等价成功结果，步骤数为 0。
- 允许后续替换具体求解算法或接入不同阶数求解策略。

不负责：

- 不直接修改 `CubeState`。
- 不直接驱动 3D 渲染。
- 不直接更新 UI 计数器。
- 不在“最优解”未确认前承诺数学最优。

状态 owner：

- 不拥有全局魔方状态，只消费 `CubeState` 快照并返回结果。
- 求解步骤数应来自 `SolverResult` 的动作序列长度，或由自动播放模块记录实际成功执行步数；二者关系需在实现阶段明确为可验证规则。

### 10. 错误与反馈状态模块

职责：

- 统一表达用户可理解的错误、提示和不支持状态。
- 覆盖 DESIGN 中要求的反馈场景：
  - 求解失败。
  - 当前状态无法生成解法。
  - 自动流程进行中，暂不能执行另一个流程。
  - 拖动输入不足以形成有效转动。
  - 浏览器或渲染环境不支持必要 3D 能力。
- 为 UI 控制模块提供可展示消息和控件状态。

不负责：

- 不吞掉底层错误导致静默失败。
- 不自行修复 `CubeState`。
- 不改变动作序列。

状态 owner：

- 当前错误/提示状态由应用控制层或专门反馈状态容器持有。
- 错误状态不得使魔方进入视觉与逻辑不一致状态。

## 模块通信

整体通信采用单向协调模式：

用户输入或 UI 事件 → 应用控制层 → 领域模块/序列模块/求解器 → 自动播放与动画调度 → 3D 渲染动画 → 动画完成回调 → 魔方逻辑状态提交 → UI 状态刷新

### 1. 初始化通信

1. UI 控制模块请求初始化。
2. 应用控制层设置默认 `CubeSize = 3`。
3. 魔方逻辑状态模型创建 3 阶完成态 `CubeState`。
4. 3D 渲染模块根据 `CubeSize` 与 `CubeState` 渲染完成态魔方。
5. UI 展示：
   - 打乱步骤数 0。
   - 求解步骤数 0。
   - 速度默认值。
   - 当前运行状态 `idle`。

### 2. 右键视角控制通信

1. 输入交互识别模块接收右键按下与拖动事件。
2. 输入交互识别模块将视角变化传递给 3D 渲染与场景模块。
3. 3D 渲染与场景模块更新相机或观察角度。
4. 不调用魔方逻辑状态模型。
5. 不生成 `Move`。
6. 不更新打乱步骤数或求解步骤数。

约束：

- 视角状态不是 `CubeState` 的一部分。
- 右键拖动不得进入动作序列模块。
- 右键拖动不得影响求解输入。

### 3. 左键层转动通信

1. 输入交互识别模块接收左键按下、拖动、松开事件。
2. 输入交互识别模块调用 3D 渲染模块提供的命中测试能力，确定命中的面或小块。
3. 输入交互识别模块结合拖动方向和当前视角生成 `InteractionIntent`。
4. 应用控制层检查当前 `RunMode`：
   - 若为 `idle`，继续处理。
   - 若处于自动打乱、自动求解或动画中，则拒绝手动转动并展示反馈。
5. 动作与序列模块将有效 `InteractionIntent` 转化为候选 `Move`。
6. 魔方逻辑状态模型校验 `Move` 对当前 `CubeSize` 和 `CubeState` 是否合法。
7. 自动播放与动画调度模块以单步动画事务执行该 `Move`。
8. 3D 渲染模块播放层旋转动画。
9. 动画完成后，魔方逻辑状态模型提交该 `Move`，生成新的 `CubeState`。
10. 3D 渲染模块根据新 `CubeState` 对齐最终视觉状态。

约束：

- 无效拖动不生成 `Move`，不改变 `CubeState`。
- 手动层转动不计入自动打乱步骤数。
- 手动层转动不计入求解步骤数。
- 输入模块不得直接改写 `CubeState`。

### 4. 随机打乱通信

1. 用户点击“随机打乱魔方”按钮。
2. UI 控制模块将事件交给应用控制层。
3. 应用控制层检查 `RunMode`：
   - `idle` 时允许开始。
   - 非 `idle` 时拒绝并展示“自动流程进行中”等反馈。
4. 应用控制层请求随机打乱模块生成 20 步合法 `MoveSequence`。
5. 随机打乱模块返回标记为 `scramble` 的动作序列。
6. 应用控制层将序列交给自动播放与动画调度模块。
7. 自动播放与动画调度模块设置 `RunMode = scrambling`，并将打乱步骤数重置为 0。
8. 每执行一步：
   - 调用 3D 渲染模块播放动画。
   - 动画完成后调用魔方逻辑状态模型提交 `Move`。
   - 打乱步骤计数递增。
   - UI 展示最新计数。
9. 20 步执行完成后：
   - `RunMode` 回到 `idle`。
   - 打乱步骤数为 20。
   - 魔方处于打乱态。
   - 清空当前动作队列。

约束：

- 自动打乱固定 20 步。
- 自动打乱期间默认禁止再次打乱、求解、手动层转动和阶数切换。
- 速度参数影响每步动画速度或间隔，不改变 20 步序列本身。
- 若随机序列极端情况下回到完成态，应在生成规则中避免等价无效打乱或重新生成，以满足可感知打乱的验收口径。

### 5. 自动求解通信

1. 用户点击“求解”按钮。
2. UI 控制模块将事件交给应用控制层。
3. 应用控制层检查 `RunMode`：
   - `idle` 时允许开始。
   - 非 `idle` 时拒绝并展示反馈。
4. 应用控制层从魔方逻辑状态模型获取当前 `CubeState` 快照和 `CubeSize`。
5. 应用控制层构造 `SolverRequest` 并调用求解器/求解适配器模块。
6. 求解器返回 `SolverResult`：
   - 成功：包含可执行 `MoveSequence` 和步骤数。
   - 已完成：返回空序列，步骤数 0。
   - 失败或不支持：返回可展示原因。
7. 若求解失败或不支持：
   - 应用控制层更新错误与反馈状态。
   - 不启动动画队列。
   - `CubeState` 保持一致。
8. 若求解成功：
   - 应用控制层将求解序列交给自动播放与动画调度模块。
   - 自动播放模块设置 `RunMode = solving`。
   - 求解步骤数按 `SolverResult` 的动作序列长度展示，或按实际成功执行进度更新；实现阶段必须固定一种可验证口径。
9. 自动播放模块逐步执行求解动作：
   - 每步播放动画。
   - 每步完成后提交逻辑状态。
   - 更新执行进度。
10. 全部完成后：
    - 魔方逻辑状态应为完成态。
    - `RunMode` 回到 `idle`。
    - UI 展示最终求解步骤数。

约束：

- 求解器不得直接修改 `CubeState`。
- 求解器不得直接驱动渲染动画。
- 求解失败不得让视觉状态和逻辑状态不一致。
- “最优解”在用户补充定义前只能作为 `SolverResult` 中的能力状态表达，不得标记为已满足。

### 6. 速度控制通信

1. 用户拖动速度滚动条。
2. UI 控制模块将新速度值交给应用控制层。
3. 应用控制层校验速度范围并更新自动播放参数。
4. 自动播放与动画调度模块读取最新速度参数。
5. 后续尚未执行的自动转动尽量按新速度生效。

约束：

- 速度只影响自动打乱和自动求解播放速度。
- 速度不改变 `MoveSequence`。
- 速度不改变求解结果。
- 速度不改变 `CubeState`。
- 自动流程中调整速度不得导致当前动画中断、卡死或状态不同步。
- 速度最小值、最大值和默认值需在实现阶段结合体验设定，并保证可感知且不会导致异常。

### 7. 阶数切换通信

1. 用户选择 3 阶、4 阶或 5 阶。
2. UI 控制模块将选择事件交给应用控制层。
3. 应用控制层检查 `RunMode`：
   - `idle` 时允许切换。
   - 自动打乱或自动求解中默认禁用切换。
4. 应用控制层设置新的 `CubeSize`。
5. 魔方逻辑状态模型重建对应阶数完成态 `CubeState`。
6. 自动播放与动画调度模块清空动作队列和临时动画状态。
7. UI 状态重置：
   - 打乱步骤数重置为 0。
   - 求解步骤数重置为 0。
   - 错误状态按需要清空或更新。
8. 3D 渲染模块销毁或重建对应阶数的魔方视觉对象。
9. 页面展示新阶数完成态魔方。

约束：

- 切换阶数后不保留上一阶魔方内部状态。
- 切换阶数后不继续播放上一阶动画。
- 计数器默认重置，但该规则仍标记为待用户确认产品规则。
- 当前默认不支持自动流程中取消并切换；若后续要求支持，需要扩展状态机和反馈规则。

## 数据流

### 1. 权威状态与派生状态

#### 权威状态

- `CubeState`
  - owner：魔方逻辑状态模型模块。
  - 含义：当前魔方每个面、贴纸或小块颜色排列的逻辑事实。
  - 用途：判断完成态、应用转动、提供求解输入。
  - 约束：不得由 UI、渲染模块、输入模块或求解器直接改写。

- `CubeSize`
  - owner：应用控制层或应用状态容器。
  - 合法值：3、4、5。
  - 用途：决定魔方结构、可执行 move set、渲染对象数量、求解输入和阶数切换逻辑。
  - 约束：不支持其他阶数，除非后续需求变更。

- `RunMode` / `PlaybackState`
  - owner：应用控制层与自动播放模块边界。
  - 建议状态：
    - `idle`
    - `manual-turning`
    - `scrambling`
    - `solving`
    - `animating`
    - `switching-size`
    - `error`
  - 用途：控制按钮可用性、防止流程重叠、支撑错误反馈。
  - 约束：同一时间只能有一个会改变魔方状态的流程运行。

- 当前动作队列与执行进度
  - owner：自动播放与动画调度模块。
  - 用途：驱动自动打乱、自动求解、单步手动转动动画。
  - 约束：流程结束后清空；阶数切换时清空。

- 打乱步骤数
  - owner：自动播放与动画调度模块或流程状态容器。
  - 来源：标记为 `scramble` 的 20 步队列实际完成进度。
  - 约束：手动转动不计入；自动打乱完成后应为 20。

- 求解步骤数
  - owner：自动播放与动画调度模块或流程状态容器。
  - 来源：`SolverResult` 动作序列长度或实际成功执行的求解动作数。
  - 约束：必须在实现阶段固定展示口径；手动转动不直接计入。

- 自动转动速度
  - owner：应用状态容器或自动播放配置。
  - 来源：速度滚动条。
  - 用途：影响自动打乱和自动求解动画时长或播放间隔。
  - 约束：不得改变动作序列和逻辑结果。

#### 派生状态

- 3D 视觉状态
  - owner：3D 渲染与场景模块。
  - 来源：`CubeState`、`CubeSize` 和当前动画进度。
  - 约束：不是权威业务状态。

- 视角状态
  - owner：3D 渲染与场景模块或输入交互识别模块。
  - 来源：右键拖动。
  - 约束：不参与求解、不影响计数、不改变 `CubeState`。

- 手动拖动临时状态
  - owner：输入交互识别模块。
  - 来源：左键按下、拖动、松开过程。
  - 约束：只在本次交互生命周期内有效，不作为长期业务状态保存。

- 错误与提示展示状态
  - owner：应用控制层或错误与反馈状态模块。
  - 来源：求解失败、流程冲突、无效输入、渲染不支持等。
  - 约束：必须可被 UI 展示，不得静默失败。

### 2. 核心 shared contract

以下为内部 public contract，跨模块依赖时只能依赖这些公开契约，不得依赖其他模块的 internal/private 实现。

#### `CubeSize`

- owner：应用控制层与魔方逻辑状态模型模块共同维护语义，架构层定义公共契约。
- 合法值：3、4、5。
- 兼容策略：
  - 当前不得扩展到其他阶数。
  - 若未来支持更多阶数，属于产品范围和公共契约变更，需要 ADR。
- ADR：当前固定 3/4/5 来自 DESIGN，不需单独 ADR；扩展阶数需要 ADR。

#### `CubeState`

- owner：魔方逻辑状态模型模块。
- 含义：不同阶数魔方的逻辑状态表达，必须能判断完成态、应用转动、提供求解输入。
- 兼容策略：
  - 状态结构应支持 3/4/5 阶统一抽象。
  - 后续字段扩展应保持向后兼容。
  - 改变贴纸排列语义、坐标系、序列化格式或完成态判断规则属于破坏性变更。
- ADR：状态模型格式需要 ADR 候选记录，尤其是选择贴纸模型、小块模型或其他内部表达时。

#### `Move`

- owner：动作与序列模块。
- 含义：一次层转动动作。
- 建议包含语义：
  - 转动轴。
  - 层索引。
  - 转动方向。
  - 转动单位，例如 quarter-turn、half-turn 或角度。
  - 适用阶数或 move set 约束。
- 兼容策略：
  - 新增字段应保持默认语义。
  - 改变方向定义、层索引基准、计数规则或转动单位属于破坏性变更。
- ADR：`Move` contract 和动作计数规则需要 ADR 候选记录。

#### `MoveSequence`

- owner：动作与序列模块，执行时由自动播放模块持有队列实例。
- 含义：有序 `Move` 列表，用于手动单步、随机打乱和自动求解。
- 兼容策略：
  - 允许增加元数据，例如来源 `manual` / `scramble` / `solve`。
  - 不得改变已有序列顺序语义。
- ADR：若引入序列压缩、规范化或异步生成策略，需要 ADR 候选。

#### `AnimationCommand`

- owner：自动播放与动画调度模块，3D 渲染模块消费。
- 含义：将逻辑 `Move` 转化为渲染可执行的动画指令。
- 兼容策略：
  - 可以扩展动画时长、缓动函数、播放速度等展示参数。
  - 不得让动画命令成为新的业务状态事实源。
- ADR：渲染动画协议和 3D 引擎选择需要 ADR 候选。

#### `InteractionIntent`

- owner：输入交互识别模块。
- 含义：左键拖动产生的候选操作意图，随后由动作与序列模块和魔方逻辑状态模型校验为合法 `Move`。
- 兼容策略：
  - 可扩展命中面、拖动向量、候选轴、置信度等字段。
  - 不得绕过校验直接提交状态。
- ADR：复杂输入识别策略通常不需要 ADR；若引入触屏、手势预测或跨设备输入模型，需要 ADR 或需求变更。

#### `SolverRequest`

- owner：求解器/求解适配器模块。
- 含义：求解器输入。
- 建议包含：
  - `CubeSize`
  - `CubeState` 快照
  - 可用 move set
  - 最优性要求或能力要求
  - 可选耗时限制
- 兼容策略：
  - 可以新增能力约束字段。
  - 不得改变 `CubeState` 快照只读语义。
- ADR：求解器输入协议和最优性定义需要 ADR 候选。

#### `SolverResult`

- owner：求解器/求解适配器模块。
- 含义：求解器输出。
- 必须能表达：
  - 成功。
  - 已完成。
  - 失败。
  - 不支持。
  - 动作序列。
  - 步骤数。
  - 是否保证最优或最优性状态。
  - 失败原因或不支持原因。
- 兼容策略：
  - 可以新增算法信息、耗时、能力说明等元数据。
  - 不得把未确认的“最优解”默认标记为已保证。
- ADR：求解算法策略、最优性定义和 3/4/5 阶能力边界需要 ADR 候选。

#### `RunMode` / `PlaybackState`

- owner：应用控制层与自动播放模块。
- 含义：当前流程状态，用于防止并发冲突和驱动 UI 禁用状态。
- 建议状态：
  - `idle`
  - `manual-turning`
  - `scrambling`
  - `solving`
  - `animating`
  - `switching-size`
  - `error`
- 兼容策略：
  - 可新增状态，但必须定义与按钮可用性、动作队列、计数更新的关系。
  - 改变自动流程互斥语义属于破坏性变更。
- ADR：如果后续支持中断、重启、队列或取消回滚，需要 ADR 候选。

#### `ErrorFeedback`

- owner：错误与反馈状态模块。
- 含义：用户可理解的失败、不支持或暂不可操作提示。
- 建议覆盖：
  - 求解失败。
  - 求解不支持当前阶数或当前状态。
  - 自动流程进行中。
  - 渲染能力不足。
  - 无效拖动。
- 兼容策略：
  - 可新增错误码或文案键。
  - 不应暴露敏感信息或不可理解的内部异常。
- ADR：若后续引入远程错误上报或用户数据收集，需要隐私与安全 ADR。

### 3. 数据流事务规则

#### 手动转动事务

1. 输入模块识别候选意图。
2. 应用控制层检查当前流程状态。
3. 动作模块生成 `Move`。
4. 状态模型校验合法性。
5. 动画调度模块执行单步动画。
6. 动画完成后状态模型提交 `Move`。
7. 渲染模块与新 `CubeState` 对齐。

事务约束：

- 动画未完成时不得提交多个冲突 `Move`。
- 如果动画失败，应保持或恢复到一致的 `CubeState`。
- 无效拖动不进入事务。

#### 自动打乱事务

1. 生成 20 步 `MoveSequence`。
2. 设置 `RunMode = scrambling`。
3. 逐步执行动画和状态提交。
4. 每步完成后更新打乱步骤数。
5. 第 20 步完成后设置计数为 20 并回到 `idle`。

事务约束：

- 不允许重入打乱。
- 不允许同时求解。
- 不允许手动层转动。
- 不允许阶数切换。
- 速度调整只影响后续播放参数。

#### 自动求解事务

1. 基于当前 `CubeState` 构造 `SolverRequest`。
2. 获取 `SolverResult`。
3. 失败或不支持时进入反馈流程，不改变 `CubeState`。
4. 成功时逐步执行求解序列。
5. 全部完成后确认 `CubeState` 为完成态。
6. 展示最终求解步骤数。

事务约束：

- 求解器结果必须可执行；若执行中发现非法 `Move`，应停止并反馈错误。
- 求解失败不得破坏当前状态。
- 求解完成态是验收关键条件。
- “最优性”只能按 `SolverResult` 明确能力字段展示，不得默认宣称。

#### 阶数切换事务

1. 仅在 `idle` 状态允许。
2. 设置新的 `CubeSize`。
3. 重建完成态 `CubeState`。
4. 清空动作队列和动画状态。
5. 重置计数器。
6. 重建 3D 表现。

事务约束：

- 不保留上一阶状态。
- 不继续播放上一阶动画。
- 默认计数器重置仍需用户确认。

## 风险与约束

### 1. “最优解”定义缺失

来源：DESIGN 明确说明 IDEA 要求“以最优解解开”，但未定义“最优解”的度量、范围和各阶魔方算法要求。

风险：

- 未明确使用 quarter-turn metric、face-turn metric、slice-turn metric 或其他计数方式。
- 未明确 3 阶、4 阶、5 阶是否都要求全局最短步数。
- 未明确求解耗时上限、浏览器主线程卡顿容忍度。
- 4 阶和 5 阶严格最优求解在浏览器中可能存在显著性能和可行性风险。
- 若直接承诺“最优解”，后续验收不可控。

架构处理：

- 求解器只定义 `SolverRequest` / `SolverResult` contract。
- `SolverResult` 必须能表达是否保证最优、非最优、降级、失败、不支持和原因。
- 不在 MSC 中选择具体求解算法。
- 不承诺所有阶数均可数学最优。

待确认：

- “最优解”的精确定义和计数度量。
- 是否 3、4、5 阶都要求严格最优。
- 是否接受先交付可复原但非最优的求解能力。
- 可接受的最大求解耗时。
- 求解失败或暂不支持某阶数时的用户反馈。

ADR：

- 求解器最优性定义需要 ADR。
- 求解算法策略需要 ADR。
- 如果求解计算需要 Web Worker 或外部库，需要 ADR。

### 2. 3D 渲染状态与逻辑状态不一致风险

来源：DESIGN 要求动画完成后视觉状态与逻辑状态一致。

风险：

- 如果渲染模块成为事实状态源，会导致计数、求解输入和完成态判断不可审计。
- 如果动画完成前后提交顺序不清晰，会出现视觉已经转动但逻辑未更新，或逻辑已更新但视觉未完成的问题。

架构处理：

- `CubeState` 由魔方逻辑状态模型模块唯一拥有。
- 渲染模块只消费 `CubeState` 与动画进度。
- 所有改变魔方状态的动作必须通过 `Move` 和动画事务提交。
- 动画完成后再提交或对齐逻辑状态，具体事务策略需实现阶段固定并测试。

ADR：

- 状态模型格式、动画提交策略、3D 渲染库选择均为 ADR 候选。

### 3. 自动流程并发风险

来源：DESIGN 要求自动流程期间避免重复点击造成重叠，并默认不要求中断能力。

风险：

- 打乱中再次打乱可能导致队列叠加。
- 打乱中求解可能导致状态与求解输入不一致。
- 求解中手动转动可能导致求解序列失效。
- 自动流程中切换阶数可能导致上一阶动画继续作用到新阶数。

架构处理：

- 默认 `RunMode` 互斥。
- 自动打乱和自动求解不可中断。
- 自动流程期间禁用重复打乱、重复求解、手动层转动和阶数切换。
- 速度调整可以继续作为播放参数生效，但不改变动作队列。

待确认：

- 是否需要中断、重启、取消或队列能力。
- 若需要取消，取消后的状态恢复规则和用户反馈。

ADR：

- 若后续引入中断、重启、取消或队列，需要 ADR。

### 4. 阶数切换规则仍需产品确认

来源：DESIGN 建议切换阶数后计数器重置，但表述为默认建议或按确认后的产品规则处理。

风险：

- 如果后续要求保留计数或历史结果，需要调整 UI 展示和状态模型。
- 自动流程中切换阶数若不禁止或取消，会破坏状态一致性。

架构处理：

- 默认仅在 `idle` 状态允许切换阶数。
- 切换后重建对应阶数完成态。
- 清空动作队列和动画状态。
- 打乱步骤数和求解步骤数默认重置为 0。
- 不保留上一阶魔方内部状态。

待确认：

- 切换阶数后两个计数器是否一律重置。
- 自动流程中是否只禁用切换，还是提供取消与重置机制。

### 5. 多阶魔方统一抽象风险

来源：DESIGN 要求支持 3 阶、4 阶、5 阶。

风险：

- 若 `CubeState` 或 `Move` 按 3 阶硬编码，后续 4 阶、5 阶会返工。
- 4 阶、5 阶中心块、棱块和可转动层规则更复杂。
- 求解能力在不同阶数上差异明显。

架构处理：

- `CubeSize` 从第一阶段即限制并覆盖 3、4、5。
- `CubeState` 和 `Move` 必须以阶数参数化设计。
- 求解器 contract 必须允许表达按阶数支持能力、不支持原因和降级策略。
- 不扩展到 2 阶、6 阶或非标准魔方，除非需求变更。

ADR：

- `CubeState` 表达方式和 `Move` 语义需要 ADR 候选。

### 6. 左键拖动识别复杂度风险

来源：DESIGN 要求根据点击位置、拖动方向和当前视角判断目标层与转动方向，并且无效拖动不能误触发。

风险：

- 输入识别依赖渲染命中、相机视角、拖动方向和层规则，容易与渲染模块耦合。
- 过度复杂的自然手势预测可能扩大范围。
- 无效拖动边界若不可测试，会影响验收。

架构处理：

- 输入模块只生成 `InteractionIntent` 或候选 `Move`。
- 状态模型和动作模块负责合法性校验。
- 输入模块不得直接更新 `CubeState`。
- 当前只支持鼠标左键和右键，不默认支持触屏或移动端手势。

待确认：

- 是否需要更复杂的手势预测、移动端触摸或键盘替代操作；当前 DESIGN 不要求。

### 7. 性能风险

来源：DESIGN 非功能要求指出浏览器中应保持可交互，5 阶对象数量增加时仍应基本可用，复杂求解不应长时间阻塞主界面。

风险：

- 5 阶魔方视觉对象数量更多，渲染和动画可能卡顿。
- 计数更新或状态计算若每帧全量重建，可能影响动画平滑。
- 求解器若在主线程执行复杂计算，可能阻塞 UI。

架构处理：

- 渲染模块应避免不必要的每帧全量重建。
- 自动播放模块应把状态提交与渲染动画解耦为可控事务。
- 求解器 contract 允许后续引入异步执行、失败、超时或不支持状态。
- 主线程 / Web Worker 边界由后续技术方案决定。

待确认：

- 目标 FPS、最大求解耗时、最低设备性能基线。
- 是否允许使用 Web Worker。
- 是否允许引入第三方求解库或 3D 库。

ADR：

- 主线程 / Worker 边界需要 ADR 候选。
- 第三方 3D 库或求解库选择需要 ADR。

### 8. 技术栈与第三方依赖未指定

来源：DESIGN 明确当前不指定前端框架、3D 渲染库、求解算法或部署方式。

风险：

- 不同 3D 引擎对命中测试、动画控制、性能优化和测试方式影响较大。
- 第三方求解库可能影响包体、兼容性、维护性和算法能力。
- 若过早绑定具体库，会限制后续切片和替换。

架构处理：

- MSC 不指定具体前端框架、3D 库或求解库。
- 渲染模块和求解器模块以 adapter 边界隔离具体实现。
- 引入第三方依赖时需说明用途、替代方案、包体或性能影响、浏览器兼容性和维护风险。

ADR：

- 3D 渲染库选择需要 ADR。
- 求解库或算法选择需要 ADR。

### 9. 不做范围约束

来源：DESIGN 不做范围。

当前架构不得默认引入：

- 用户账号、登录、注册或个人数据同步。
- 魔方状态云端存储。
- 排行榜、计时比赛、成绩历史或社交分享。
- 教学模式、公式说明、逐步讲解或提示系统。
- 移动端触摸手势明确支持。
- 实体魔方设备连接。
- 多人协作或实时联机。
- 主题皮肤、材质编辑或自定义颜色。
- 3/4/5 阶以外的其他阶数。
- 非标准魔方形态。
- 离线 PWA、桌面安装或原生 App 能力。
- 指定必须使用某个 3D 引擎、前端框架或求解算法。
- 在“最优解”定义未确认前承诺具体算法复杂度、最大求解时间或数学最优性。

若后续新增上述范围，应进入需求变更和架构复核。

### 10. 兼容性、迁移与回滚约束

来源：DESIGN 非功能要求。

兼容性：

- 目标为现代浏览器环境。
- 需要支持浏览器 3D 渲染能力。
- 不承诺旧版浏览器。
- 不承诺移动端触摸体验。
- 浏览器不支持必要能力时应展示错误或降级提示。

迁移：

- 当前无后端数据、数据库、用户存档，不涉及数据迁移。
- 若后续引入持久化魔方状态、用户记录或配置保存，需要定义数据结构版本和迁移策略。

回滚：

- 某个阶数、求解器或渲染能力不可用时，应能通过禁用入口或回退到稳定版本避免影响整体页面。
- 自动流程失败时，应停止动画并保持或恢复到一致的魔方逻辑状态。
- 当前不涉及数据库回滚。

### 11. 安全与隐私约束

来源：DESIGN 非功能要求。

约束：

- 当前不涉及登录、支付、权限、服务端管理或敏感操作。
- 不应在前端代码或文档中写入真实密钥、token、cookie、私钥或生产账号。
- 当前不收集、上传或存储用户个人信息。
- 不默认记录用户操作历史到远程服务。
- 若后续加入分析统计、错误上报或用户数据存储，需要重新补充隐私说明和用户告知机制。

### 12. 可访问性约束

来源：DESIGN 非功能要求。

约束：

- 关键按钮和控件应有可理解文本标签：
  - “随机打乱”
  - “求解”
  - “速度”
  - “3 阶”
  - “4 阶”
  - “5 阶”
- 步骤计数器应使用文本展示，不能仅依赖颜色或动画。
- 当前阶段不强制键盘替代操作、屏幕阅读器完整支持和移动端触摸交互。
- 若后续面向更广泛用户发布，需要补充键盘操作、焦点管理和 ARIA 描述。
